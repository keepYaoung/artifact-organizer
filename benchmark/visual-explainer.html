Artifact Organizer codebase map

Prompt in. Structured HTML out.

This repo is a catalog-driven renderer. The prompt layer shapes a semantic JSON envelope. The Node renderer validates it, walks the tree, and emits a single self-contained HTML file with a sibling JSON artifact.

Core source: plugins/hyperscribe Mirror copies live in skills/hyperscribe* Runtime: Node 20 + node --test Outputs: .html + sibling .json

System flow

How a prompt becomes a page

The pipeline is small on purpose. Validation happens before rendering, and rendering happens before any browser sees broken structure.

Intent layer

User prompt -> command or skill -> envelope JSON

The model does not write HTML directly. It emits a semantic envelope with a root component, catalog, and parts.

  • General mode targets artifact-organizer/Page.
  • Slide mode targets artifact-organizer/SlideDeck.
Validation layer

scripts/lib/schema.mjs checks the contract

Missing fields, type mismatches, bad enums, unknown components, and root errors are caught before the browser ever loads the result.

Tree layer

scripts/lib/tree.mjs walks the component graph

Registry-backed renderers turn nodes into HTML strings. Container components recurse. Leaf components return their markup directly.

  • Dashboard rerenders its panel children explicitly.
  • The renderer stays close to pure string generation.
Delivery layer

Theme, CSS, and interactive JS are stitched together

The final artifact is assembled from theme CSS, base CSS, component CSS, body HTML, toggle markup, and assets/interactive.js.

  • Used component CSS only is inlined.
  • share.sh can publish the finished HTML.
1. Prompt and skill layer

Agents decide the shape.

Command docs and skill prompts guide the LLM toward the right envelope and root mode. This is where layout intent and content structure are negotiated.

2. Renderer layer

Node turns contracts into markup.

render.mjs, schema.mjs, tree.mjs, and the component files do the deterministic work. The codebase acts more like a UI DSL compiler than a web app.

3. Output and sharing layer

HTML lands in the browser, JSON stays beside it.

The repository preserves both the rendered result and the input envelope, which makes debugging and review much easier.

Repository map

Where the important pieces live

The same idea repeats in three forms: product source, mirrored skills, and tests that lock the contract down.

Product source

plugins/hyperscribe

The actual plugin package. It holds renderer code, assets, catalog, commands, and share tooling.

Skill mirrors

skills/hyperscribe*

Deployment-facing copies for general, slides, diff, and share modes. They wrap the same renderer.

Tests

tests/components, tests/lib, tests/*.test.mjs

Component snapshots, schema validation, tree walking, preference parsing, and HTML helper coverage.

Docs and tools

README.md, docs/plans, tools/build-catalog-md.mjs

Reference material and regeneration scripts. The catalog is treated as the source of truth.

Key files

The files that explain the codebase fastest

These are the shortest routes to the system's behavior, its contract, and its seams.

plugins/hyperscribe/scripts/render.mjs Main entry

Builds the registry, validates input, loads theme CSS, and assembles the final document.

plugins/hyperscribe/spec/catalog.json API contract

Defines the envelope and component prop schemas. Everything else orbits this file.

plugins/hyperscribe/scripts/lib/schema.mjs Guardrail

Validates shape, enums, patterns, and root component choice before render time.

plugins/hyperscribe/scripts/lib/tree.mjs Walker

Recursively invokes registered renderers and handles the component tree traversal.

plugins/hyperscribe/scripts/components/*.mjs SSR leaflets

Thin component renderers. Most are direct prop-to-HTML string functions.

plugins/hyperscribe/assets/base.css and assets/components/*.css Style assembly

Base design system plus per-component styles, with only used component CSS inlined.

One line that matters

spec/catalog.json is the contract. The component renderers are just one implementation of that contract, and the tests prove they still agree.

  • Read the catalog first when adding or changing a component.
  • Then update the renderer, CSS, and tests in that order.
  • Rebuild the catalog docs from the JSON, not the other way around.
Tradeoffs

What the repo optimizes for

The design is deliberately lean. It favors clarity, repeatability, and easy debugging over heavyweight runtime machinery.

Deliberately simple

  • No build step. Node 20 and node --test are enough.
  • No state-heavy client app. Most renderers are pure string functions.
  • Validation happens before browser rendering, which keeps failures crisp.
  • The system keeps both HTML output and the envelope that created it.

Not fully offline everywhere

  • Mermaid and Chart can still load from jsDelivr.
  • Image is closer to true self-contained behavior when assets are local.
  • The HTML is portable, but some visual features still depend on runtime network access.
